# v0.1.1 Release Readiness

## 1. Release purpose
v0.1.1 is a post-release hardening/docs/policy follow-up to the public v0.1.0 release gate. It is intended to improve project safety, OSS onboarding clarity, and repository operating discipline without changing application behavior.

## 2. Completed changes since v0.1.0
- GitHub Actions CI baseline added for go vet, go test, docker compose config -q, and docker compose build app
- CI follow-up added to prepare a safe .env.local for Docker Compose in CI
- Branch protection / ruleset setup was completed
- Public clone path/docs fix was completed for the public repo slug
- Setup wizard UX review note was added
- Setup docs/copy were aligned with the current wizard entry behavior
- Historical tag policy was documented
- Branding / repository naming policy was documented

## 3. Validation status
- This release line is documentation, CI, and policy hardening only
- No application behavior change is intended in v0.1.1
- CI baseline exists in the repository and is the main release-safety improvement for future PR validation
- Public onboarding docs now point to the correct public repository path and better match the current setup entry flow
- This note does not rerun build/test itself; it summarizes the completed hardening work already landed on master

## 4. Known limitations
- v0.1.1 does not change setup/auth/runtime/API/Docker/DB behavior
- Setup wizard trust gaps were reduced in docs, but not fully solved in-product
- Branch protection checks were configured, but ruleset/check behavior should still be monitored over time
- Historical non-v tags remain in place by policy for now
- Repository slug and public product name remain intentionally different for now

## 5. What is intentionally not included
- No setup flow rewrite
- No auth/session redesign
- No runtime bootstrap redesign
- No repository rename
- No historical tag deletion
- No new feature scope beyond hardening/docs/policy work

## 6. Release recommendation
Release now.

Reasoning:
- v0.1.1 is small, well-scoped, and consistent with a hardening/docs/policy release
- the completed changes reduce OSS onboarding and repository operation risk without expanding runtime scope
- there is no evidence in this pass that v0.1.1 should be held for additional product behavior work
