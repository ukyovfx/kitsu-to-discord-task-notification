# KitsuSync v0.3.0 Release Readiness

## Summary
v0.3.0 focused on reliability and operator safety hardening rather than feature expansion. This phase addressed three operational risk areas: log secret exposure, setup endpoint side effects, and partial-failure cleanup during Discord provisioning.

## Completed Hardening Work

### 1. Log Secret Redaction
- PR #59 (merge commit: `ccaf300`)
- Redaction is wired through both stdout and file log paths so console and `logs/all-levels.log` are covered.
- Redaction behavior reads env values at call time, so runtime `os.Setenv()` updates are still redacted.
- slog handler wiring was adjusted to the supported writer path to preserve existing visible log format/destinations while applying redaction.
- Partial-write handling in the redacting writer was hardened and validated with targeted tests.

### 2. Side-effect-free Discord Connection Testing
- PR #60 (merge commit: `5ac4a7c`)
- `/api/setup/test-discord` now remains validation-only and no longer mutates runtime env or persisted setup state.
- Bot token and guild persistence semantics were clarified in admin copy/docs to reduce operator misunderstanding.
- This lowers risk of accidental live configuration mutation during “test” actions.

### 3. DB Failure Discord Cleanup
- PR #62 (merge commit: `fd3f5fb`)
- Added best-effort cleanup when Discord resources are created but the subsequent DB transaction fails.
- Cleanup scope is limited to request-scoped resources created in the same setup request (created channels/category only).
- Success path behavior is unchanged.
- Cleanup is designed as idempotent/best-effort: delete failures are tolerated and logged for operator follow-up.

## Validation Performed
- `go test ./src/setup/... -count=1` (container)
- `go test ./src/...` (container)
- `docker compose build app`

## Remaining Known Gaps
- No full rollback orchestration across all setup side effects.
- No persistent tracking table/system for orphaned external resources.
- Repair and reconciliation UI remains limited.
- CI/test depth is still limited compared with full production-hardening baselines.

## Non-goals
v0.3.0 did not attempt:
- release/tag creation automation or release publication.
- full transactional redesign of setup/provisioning.
- encrypted secret persistence redesign.
- broad setup UX restructuring beyond scoped safety fixes.
- unrelated runtime/API/DB behavior changes outside the targeted hardening issues.
