# v0.1.3 Release Readiness

## 1. Release purpose
v0.1.3 is a small docs cleanup and docs route consistency release. Its purpose is to remove an orphaned docs artifact and align active in-app docs routing with the paths already referenced by the repository and admin UI, without redesigning the docs architecture or changing non-docs behavior.

## 2. Completed changes since v0.1.2
- Removed the orphaned `src/docs.html` artifact that was no longer referenced by the active docs flow
- Kept the active root `docs.html` entry point in place
- Kept `site.jsx`, `build_site.py`, and `diagrams/*.jsx` in place
- Kept the existing `/docs` and `/site.jsx` routes in place for compatibility
- Added `/bot/docs`, `/bot/docs/`, and `/bot/docs/site.jsx` route aliases so active in-app docs links match Go routing behavior

## 3. Validation status
- This release line is limited to docs cleanup and docs route consistency work
- No setup/auth/runtime/API/Docker/DB behavior change is intended
- #30 validation confirmed `git diff --check` passed, no remaining references to `src/docs.html` were found, and the active root `docs.html` entry point was kept
- #32 validation confirmed `/bot/docs` and `docs.html` grep checks were reviewed, and `docker compose config -q` plus `docker compose build app` passed after the route alias change
- This readiness note does not claim local `go test` or `go vet` execution
- This readiness note does not claim production maturity

## 4. Known limitations
- v0.1.3 does not redesign the docs architecture
- Markdown docs still do not replace the HTML docs because they serve different readers
- The active docs page still depends on the existing static `docs.html` plus `site.jsx` structure
- This release improves route consistency, but does not broaden release claims beyond the validated cleanup and alias work

## 5. What is intentionally not included
- No setup flow rewrite
- No auth/session change
- No runtime/bootstrap change
- No API/schema change
- No Docker behavior change beyond previously validated app build compatibility
- No DB change
- No docs architecture replacement or Markdown-for-HTML swap
- No additional cleanup work beyond the orphaned artifact removal and docs route consistency fix
